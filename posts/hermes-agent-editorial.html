如果你只是把 `nousresearch/hermes-agent` 看成另一個 agent 工具，那你很可能低估它了。

因為這個 repo 現在真正有意思的地方，不只是 stars 很高，也不只是 release 很勤，而是它已經明顯超過「單一 agent 工具」的範圍，開始長成一個通用 agent platform。

從數字看，它已經是超大規模社群專案：

  • 9 萬多 stars
  • 1.2 萬多 forks
  • issue / PR 編號已經破 11000
  • release 幾乎幾天一版

但真正值得看的，不是熱度本身，而是它正在碰什麼問題。

它在做的，不只是 CLI agent

從 README 與 repo 結構就看得出來，Hermes Agent 現在的能力面非常廣：

  • gateway
  • cron
  • skills
  • memory
  • MCP
  • subagents
  • toolsets
  • web / website
  • research / trajectory / compression
  • 多 provider / 多 channel / 多 terminal backend

它不是一個只靠 prompt 或單一對話迴圈運作的小工具，而是一個想把 agent 長期運作所需的東西盡量包起來的平台。

README 裡甚至直接把自己描述成：

  • 具備 closed learning loop
  • 支援 messaging gateway
  • 內建 cron scheduler
  • 可 delegate 與平行化
  • 可跑在 VPS、GPU cluster、serverless infra
  • 能從 OpenClaw 自動 migrate

這些訊號放在一起看，其實很清楚：

Hermes Agent 正在競爭的，不只是 agent 工具的位置，而是通用 agent runtime / platform 的位置。

真正值得追的，是平台摩擦開始浮出來了

Hermes Agent 現在最有價值的地方，不是「功能很多」，而是你開始看得到一個 agent 平台真實會卡在哪裡。

像最近的 commit、PR、issue，很能代表這點：

  • gateway transcript atomic rewrite，避免中途毀損
  • `/status` 補 model / provider 資訊
  • context compressor 要保留最後一則 user message，避免 active-task loss
  • sub-agent polling loop 出現 UnboundLocalError
  • Telegram exec approval prompt 要 escape command content
  • provider base_url resolution 出現 collision
  • `delegate_task` 環境缺 `$HOME`
  • long tool-only run 沒有 still working update

這些問題都不是表面功能沒接好而已，而是在說：

  • session lifecycle 很複雜
  • gateway 是真核心
  • sub-agent 很容易失真
  • provider abstraction 很容易裂開
  • 長任務 UX 與工具安全很難同時兼顧

也就是說，Hermes Agent 現在已經進入平台級問題的密集區。

Gateway 是最值得盯的戰場

從最近的變化看，gateway 很明顯是 Hermes Agent 的中樞之一。

你會看到很多改動都直接打在 gateway：

  • transcript rewrite
  • `/status` UX
  • channel prompt guard
  • SMTP-only outbound
  • Telegram 顯示細節
  • QQ / TTS / voice 類問題

這代表 gateway 不是附屬功能，而是：

  • session 狀態入口
  • 多平台延續層
  • delivery 層
  • approval / status / feedback UX 層

如果 agent 最後真的要變成可長期使用的系統，那 gateway 幾乎一定會是平台的核心之一。Hermes Agent 現在正好把這件事攤開來讓大家看。

Provider abstraction 也開始進入真實世界

另一個很值得看的點，是 Hermes Agent 現在支援的 provider 與模型面越來越廣。

README 直接列了一大串：

  • Nous Portal
  • OpenRouter
  • Xiaomi MiMo
  • z.ai/GLM
  • Kimi/Moonshot
  • MiniMax
  • Hugging Face
  • OpenAI
  • 自訂 endpoint

這當然很吸引人，但也會直接帶來 abstraction 複雜度。

最近一個很典型的例子，就是 issue #11059：

Cron sub-agent path collision: `/v4/v1/messages`

主 session 正常，但到 cron / delegated sub-agent 路徑時，OpenAI 與 Anthropic 的 endpoint resolution 被錯誤拼接，最後變成一個不存在的路徑。

這類問題很有代表性，因為它不是主流程不能用，而是：

  • 主流程能用
  • 子流程、子 session、delegation 一複雜就開始出現不一致

這正是平台 abstraction 最常出問題的地方。

Hermes Agent 跟 OpenClaw 放在一起看，會更有意思

如果把 Hermes Agent 跟 OpenClaw 放在一起看，對照感會很強。

Hermes Agent 比較像：

  • 超高速擴張中的大型通用 agent 平台
  • 社群驅動很強
  • 功能面極廣
  • 很多能力都想原生包進來

OpenClaw 比較像：

  • 更高可塑性的 agent runtime
  • 更容易做成個人 AI 管家 / 日常工作中樞
  • workflow / channel / cron / memory 感更直接
  • 比較適合深度客製自己的系統

也就是說，兩者雖然都在做 agent platform，但哲學有點不同。

Hermes Agent 更像大平台。OpenClaw 更像高可塑 runtime。

我現在的判斷

我對 Hermes Agent 的看法很簡單：

### 值得追蹤度：很高

因為它正在處理 agent 平台最真實的問題，而不是只做表面 demo。

### 穩定導入難度：不低

因為 release 很快、issue 很多、provider / gateway / sub-agent 複雜度都高。

### 內容價值：非常高

因為它非常適合拿來看：

  • agent platform 怎麼長大
  • gateway 怎麼成為中樞
  • sub-agent 怎麼暴露平台摩擦
  • provider abstraction 會在哪裡裂開

最後一句話

Hermes Agent 最值得追的，不只是因為它大，而是因為它正在把 agent 平台真正困難的地方暴露出來。

如果你要看未來 agent 系統怎麼從工具變成平台，這個 repo 很值得持續盯。
