最近在追的 3 個 GitHub repo：Onyx、Harness、Humanizer-zh-TW

副標

從 AI 平台、工程治理到文字編輯，我最近看到三種很不一樣的開源節奏

---

### 摘要

最近我整理了三個正在追蹤的 GitHub repo：Onyx、Harness 與 Humanizer-zh-TW。它們分別代表 AI 平台、工程基礎設施與寫作技能包三種不同方向。這篇文章不是單純列功能，而是從它們各自解決的問題出發，看三種不同類型的開源專案，如何對應產品化、治理與內容修整。

---

最近我花了一些時間看幾個 GitHub repo。原本只是想追一下各自的近況，結果放在一起看之後，反而覺得很有意思。

因為它們幾乎代表了三種完全不同的開源節奏：

  • 一種是還在高速往前衝的 AI 產品
  • 一種是偏成熟穩定的工程平台
  • 一種則是小而聚焦、但切得很準的技能型專案

我最近追的三個 repo 是：

  • Onyx
  • Harness
  • Humanizer-zh-TW

如果只看類型，這三個東西很難被放在一起。

但如果從「它們各自正在回答什麼問題」來看，反而很適合一起聊。

Onyx：很有產品感，也很有速度感

先講 `onyx-dot-app/onyx`。

這個 repo 最明顯的特徵，就是你看得出來它還在快速長大。

它不是停在 demo 階段的 AI 專案，而是已經有很強的產品感，方向也很清楚：AI chat、RAG、企業搜尋、connector、多模型互動，幾乎都是現在 AI 平台最核心的題目。

從最近的 commit 和 release 節奏來看，Onyx 的更新密度很高。

而且它不是只修單點功能，而是同時在往好幾個面向推進，包括：

  • chat 體驗
  • 多模型介面
  • indexing 與 ingestion
  • connector 穩定性
  • multi-tenant 與監控細節

這類專案很吸引人，因為你會明顯感覺到它不是在維護一個已經完成的東西，而是在持續把產品往下一個階段推。

但這也是它最需要小心的地方。

更新快，通常同時意味著兩件事：

第一，功能進步很快。

第二，變動風險也高。

所以如果目標是研究趨勢、看產品方向、觀察 AI 平台如何從功能走向產品，Onyx 很值得追。

但如果是要拿來導入正式環境，我反而會建議保守一點，不要直接一路跟最新版，而是挑一條小版本線慢慢看。

簡單說，Onyx 給我的感覺是：

很值得觀察，但不適合太衝動地使用。

Harness：不熱鬧，但很重

第二個是 `harness/harness`。

它和 Onyx 是很不一樣的東西。

如果說 Onyx 看起來像是在快速進化中的產品，那 Harness 更像是一套成熟工程平台的骨架，現在持續往企業需求和治理細節補強。

它處理的核心不是 AI，而是更偏基礎設施的題目，例如：

  • SCM
  • CI/CD
  • branch rules
  • merge queue
  • audit logs
  • RBAC
  • artifact registry

這種專案不一定看起來最花俏，但你只要做過團隊開發流程、權限治理、CI pipeline 或內部平台，就會知道這些東西有多關鍵。

從最近 repo 裡看到的重點，也大多落在這些比較工程、比較治理的細節上：

  • commit signature verification
  • merge queue protection
  • branch rule 調整
  • audit logging
  • webhook 與 registry 邊角案例

這些改動不太會讓人有「哇，好炫」的感覺，但通常都很實際。

因為這類專案一旦做不好，最後影響的不是某個單一功能，而是整個團隊的工作節奏。

所以 Harness 給我的印象是：

它不是一個看新功能會很興奮的 repo，而是一個你真的做工程平台時，會很認真看的 repo。

它的價值不是熱鬧，而是穩定。

不是展示，而是承載。

Humanizer-zh-TW：小型 skill repo 的實用價值

第三個 repo 是 `kevintsai1202/Humanizer-zh-TW`。

和前兩個相比，這個 repo 小很多，也幾乎不算是工程型專案。

它本質上比較像一個 skill / prompt 規則包，目的很直接：幫使用者去除文字中明顯的 AI 痕跡，讓內容更自然、更像人寫的。

這類 repo 很容易一開始被低估。

因為它沒有複雜架構，也沒有大量程式碼，看起來就是一個小型技能包。

但實際讀下來，我反而覺得它碰到的是一個現在很真實、也很普遍的問題：

大家現在不一定缺寫作初稿，而是缺「把初稿改得自然」的方法。

AI 已經很會生成內容了。

但很多文字一看就還是有一種熟悉的味道：

  • 空泛的大詞太多
  • 模板式連接詞太多
  • 三段式形容太重
  • 結尾過度空泛
  • 整體語氣像簡報或新聞稿，不像真的有人在說話

Humanizer-zh-TW 的價值就在於，它不是只說「這段很 AI」，而是把常見模式拆出來，告訴你哪些地方有問題、為什麼有問題，以及通常可以怎麼改。

這種整理，對日常寫作其實很有幫助。

尤其是在 AI 已經會大量幫忙產生草稿的情況下，真正拉開差距的，常常不是生成本身，而是後面的編輯能力。

當然，這個 repo 也有很明顯的限制。

它比較像規則手冊，不是成熟產品；維護節奏偏靜態；文檔也已經開始出現一些過時訊號。

所以如果問我值不值得用，我的答案會是：

值得吸收，不建議原樣照搬。

把它當成一套規則來源、編輯框架或 skill 素材，我覺得很有價值。

但如果期待它是一個直接裝上就能無痛長期使用的工具，那可能會高估它目前的成熟度。

三個 repo，剛好代表三種不同問題

如果把這三個 repo 放在一起看，其實會很清楚它們各自卡在不同的位置：

### Onyx 在回答的是：

AI 平台怎麼快速產品化？

### Harness 在回答的是：

工程平台怎麼維持穩定、治理與流程控制？

### Humanizer-zh-TW 在回答的是：

當 AI 已經很會寫之後，人類編輯真正的價值還剩下什麼？

這也是我覺得它們值得一起看的原因。

不是因為它們是同一類產品，而是因為它們剛好從不同角度回答了「工具與人怎麼分工」這件事。

Onyx 比較像前台。

Harness 比較像地基。

Humanizer-zh-TW 則比較像最後一哩的修整。

我現在怎麼看「值得追的 repo」

以前我比較容易用 star、更新頻率、熱度去看一個 repo 值不值得追。

但最近越來越覺得，真正值得追的專案，往往不是因為它最熱門，而是因為它回答了一個你真的在意的問題。

像 Onyx 值得追，不只是因為它是 AI 專案，而是因為它讓你看到 AI 平台怎麼長成產品。

Harness 值得追，不只是因為它是工程平台，而是因為它處理的是團隊規模一大之後一定會遇到的治理問題。

Humanizer-zh-TW 值得追，也不只是因為它是中文 skill，而是因為它剛好切中了一個現在很常見但常被略過的問題：

當 AI 已經能幫你把話寫出來之後，什麼才算是真的寫好？

結語

如果只看表面，這三個 repo 很難被放在同一篇文章裡。

一個偏 AI 平台，一個偏 DevOps / SCM 基礎設施，一個偏寫作規則包。

但放在一起看之後，我反而覺得它們剛好構成了一條很完整的線：

  • 生成
  • 治理
  • 修整

這三件事，本來就是今天工具使用最重要的三個面向。

Onyx 讓我看到產品正在長大。

Harness 讓我看到系統怎麼撐住團隊。

Humanizer-zh-TW 則讓我想到，最後留下差異的，很多時候不只是工具有多強，而是人怎麼判斷、怎麼刪改、怎麼讓內容更像自己。

這大概也是我最近看 repo 最有感的一件事：

工具會越來越強，但最後拉開差距的，常常還是編輯、取捨與語氣。
