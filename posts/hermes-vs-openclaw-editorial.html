標題

Hermes Agent 與 OpenClaw：一個像大平台，一個像高可塑 runtime

副標

同樣都在做 agent system，但 Hermes Agent 與 OpenClaw 更像兩種不同的產品哲學。

---

如果把 Hermes Agent 與 OpenClaw 放在一起看，最有意思的地方不是誰功能比較多，而是它們其實代表兩種不同的 agent 平台路線。

Hermes Agent 給人的感覺很明確：

它是一個正在高速膨脹中的大型 agent platform。

OpenClaw 給人的感覺則不同：

它更像一個高可塑、工作流導向、適合深度客製的 agent runtime。

這兩種路線都很有價值，但看的角度不一樣。

Hermes Agent：大一統平台傾向很強

從 Hermes Agent 的 repo 結構、README、release 節奏與 issue / PR 動態看，它正在做的事情非常大：

  • gateway
  • skills
  • memory
  • cron
  • MCP
  • subagents
  • toolsets
  • 多 provider
  • 多 channel
  • 多 terminal backend
  • 甚至還有 RL / trajectory / compression

這種氣質很像：

想把 agent 長期運作所需的所有東西，盡可能都包進同一個平台。

它的優勢很明顯：

  • 能力很廣
  • 社群動能強
  • 平台感很重
  • 往通用 agent 作業層靠近

但代價也很明顯：

  • 複雜度高
  • 變化快
  • provider / gateway / sub-agent 邊角很多
  • 穩定性要自己控

OpenClaw：更像可塑 runtime 與個人 AI 管家底盤

OpenClaw 雖然也有很多平台元素，但整體感受比較不一樣。

它的強項在於：

  • channel 感很強
  • session 與 messaging 很直接
  • cron / memory / workspace / persona 整合順
  • 比較容易做成自己的私人 AI 管家與工作中樞

Hermes Agent 比較像在追求大平台統合。

OpenClaw 則更像在提供一個很有彈性的 runtime，讓你把自己的工作流、習慣、角色與工具黏進去。

最大差別，其實不是功能，而是哲學

Hermes Agent 的問題意識比較像：

  • agent 能不能變成通用平台
  • 所有能力能不能往同一個系統裡收斂
  • gateway / memory / skills / provider / sub-agent 能不能一次打包

OpenClaw 的問題意識則比較像：

  • agent 能不能真的融進日常工作流
  • channel / session / cron / persona 能不能被塑造成你的系統
  • 可不可以用比較低摩擦的方式做深度客製

所以如果要一句話總結：

Hermes Agent 比較像大型平台，OpenClaw 比較像高可塑 runtime。

誰比較值得追？

其實兩個都值得。

### 如果你想看未來 agent platform 怎麼長大

追 Hermes Agent 很有意思。

因為它正在暴露平台化最真實的困難：

  • gateway
  • provider abstraction
  • sub-agent orchestration
  • long-running UX
  • session continuity

### 如果你想做自己的 AI 管家 / 工作流中樞

OpenClaw 很有意思。

因為它比較容易被塑造成自己的系統，而不是只能活在官方定義裡。

最後一句話

Hermes Agent 與 OpenClaw 並不是簡單的替代關係。

它們更像是兩種不同的 agent 平台哲學：

  • 一個往大平台整合走
  • 一個往高可塑 runtime 走

如果你把這兩條路一起看，反而會更清楚未來 agent system 可能怎麼分化。
